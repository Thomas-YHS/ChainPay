# ChainPay 产品概述

> 基于 Li.Fi 的跨链薪资路由工具 · 2026年4月

---

## 1. 产品定位

**ChainPay** 是一个面向 Web3 团队的跨链薪资路由工具，以 **Li.Fi 作为核心执行引擎**，以 **智能合约作为规则执行层**。雇主在单一链上完成发薪，ChainPay 合约读取员工在链上登记的接收规则，通过 Li.Fi 自动将资产路由到员工在任意链上指定的任意 Token 组合。

产品的核心价值在于：将"员工以何种形式持有收入"这件事，从发薪流程中完全解耦出来。雇主只需按合同金额发薪，员工自己决定收什么、在哪条链上收。规则存储在链上，透明、不可篡改、无需信任平台。

---

## 2. 市场定位与差异化

### 竞品格局

现有 Web3 发薪产品均聚焦于优化雇主发薪侧的效率：

- **Request Finance**：多链批量发薪 + 财务报表工具，不做路由优化
- **Rise**：混合法币/稳定币发薪，USDC 为主，员工选择粒度粗（要不要加密）
- **Deel**：传统 HR 附加加密功能，仅支持美国，底层接 Coinbase

### 核心差异化

没有一家竞品让员工自由配置跨链多 Token 分配规则，没有人使用 Li.Fi 这类聚合路由来优化执行，也没有人将员工接收规则存储在链上实现真正的去信任化。

> **ChainPay 的差异化 = Li.Fi 的路由能力 × 员工的资产自主权 × 链上规则的透明可验证**

## 3. 系统架构

ChainPay 由四层组成：

```
前端（React + wagmi）
  ├── 雇主管理端：员工档案管理、USDC Approve、手动发薪触发
  └── 员工配置端：钱包身份验证、接收规则配置（写入链上）

ChainPay 智能合约（Solidity，部署在 Base）
  ├── 存储员工接收规则（链上透明，不可篡改）
  ├── 验证执行者身份（onlyExecutor）
  └── 调用 Li.Fi 合约执行跨链路由

后端（Go + Gin）
  ├── 员工档案管理（链下：姓名、昵称、发薪频率）
  ├── 发薪记录存储
  └── 定时任务（robfig/cron）→ 触发执行者钱包调用合约

执行者钱包（Go 后端持有，私钥存环境变量）
  └── 只能调用 ChainPay 合约，合约约束执行范围
```

---

## 4. Li.Fi 在产品中的核心角色

Li.Fi 是整个产品路由能力的技术基础。它同时聚合多个跨链桥（Stargate、Hop、Across 等）和 DEX（Uniswap、Curve 等），计算最优路径，让复杂的跨链 Token 兑换压缩成单次合约调用。

### 执行流程示例

```
雇主发出：Base 上的 USDC 5,000
         ↓
  ChainPay 合约读取员工链上规则
         ↓
    Li.Fi 路由引擎
  同时查询多个桥 + 多个 DEX
  计算最优拆分路径
         ↓
员工收到：
  40%  →  ETH  (Base)       ← Li.Fi swap
  40%  →  USDC (Arbitrum)   ← Li.Fi bridge
  20%  →  DAI  (Gnosis)     ← Li.Fi bridge + swap
```

没有 Li.Fi，实现上述路径需要自行集成每一条链的每一个桥和 DEX，工程量不可行。Li.Fi 让产品专注于规则配置体验和合约设计，而非基础设施集成。

## 5. 产品功能设计

### 雇主 / 管理端

- **员工档案管理**：录入员工姓名、昵称、钱包地址、合约金额（USDC）、发薪频率（每天 / 每周 / 每月）
- **USDC Approve**：对 ChainPay 合约授权精确发薪金额，每次发薪前执行
- **手动发薪**：选择员工 → 展示路由预览（路径、预计到账、Gas 费）→ 确认执行
- **定时自动发薪**：根据员工发薪频率，Go 后端 Cron 到期自动触发，无需雇主在线
- **发薪记录**：历史发薪的时间、金额、状态、tx hash

### 员工端

- **身份验证**：连接钱包 → 系统检查地址是否已被雇主添加
  - 未添加 → 提示联系雇主
  - 已添加 → 进入配置页
- **接收规则配置**：选择目标链、Token 类型、分配比例（最多 5 条规则，总和 = 100%），规则写入链上合约
- **规则不可修改**：设置一次后永久生效，链上存储保证透明可验证
- **自动执行**：每次发薪自动触发 Li.Fi 路由，员工无需任何手动操作

## 6. 关键设计决定

| 设计问题 | 决定 | 备注 |
|---|---|---|
| 核心执行引擎 | Li.Fi SDK + 合约 | 负责所有跨链路由和 Token 兑换 |
| 员工规则存储 | 链上（ChainPay 合约）| 透明可验证，平台无法篡改 |
| 员工档案存储 | 链下（PostgreSQL）| 姓名等非敏感信息，无需上链 |
| 定时发薪执行 | 执行者钱包调用合约 | 雇主无需在线，后端 Cron 自动触发 |
| 执行者钱包安全 | Approve 精确金额 + 私钥存环境变量 | 执行者无资产，损失上限可控 |
| Gas 费承担 | 执行者钱包支付（即公司承担）| 黑客松阶段简化处理 |
| 员工钱包绑定 | 雇主录入，员工钱包登录验证 | 无需确认链接，直接用钱包身份 |
| 规则修改 | 不支持 | 降低合约复杂度，黑客松阶段锁定 |
| 合规定位 | 工具而非支付服务商 | 合规责任在用户，早期轻装上阵 |
| 部署链 | Base（主）/ Gnosis（备）| Gas 极低，Li.Fi 支持完善 |

## 8. 待讨论事项（未来版本）

### 员工规则修改机制
当前版本规则设置后不可修改。未来可引入：
- 时间锁机制（修改请求需等待 N 天生效）
- 雇主确认机制（员工申请修改，雇主审批）

### Gas 费承担模式
当前版本由执行者钱包（公司）统一承担 Gas。未来可支持：
- 从发薪金额中自动扣除 Gas
- 员工和雇主按比例分摊

### 合约升级机制
当前版本为不可升级合约。未来生产版本需引入：
- OpenZeppelin Proxy 升级模式
- 多签治理（Gnosis Safe）

### 多链部署
当前版本仅部署在 Base。未来可扩展至：
- Arbitrum、Optimism、Polygon 等主流 L2
