# ChainPay 审计记录 v0.5 — Pure Composer 架构变更

> 审计日期：2026-04-10
> 变更范围：后端发薪路径重构（Pure Composer），前端规则读取解耦

---

## 架构变更摘要

### 变更前（v0.4）
```
ExecutePayout:
  后端 → chainPayABI.Pack("executePayout") → ChainPay 合约 → Li.Fi Diamond → 员工
```

### 变更后（v0.5）
```
ExecutePayout:
  后端 → approve USDC → LiFi Diamond
       → GET /v1/quote（fromAddress = 后端钱包）
       → 签名并广播 transactionRequest
       → LiFi 从后端钱包拉取 USDC → 路由到员工
```

**关键差异：**
- 发薪执行不再依赖 ChainPay 合约
- `ChainPayContract` 配置项不再是发薪必填（仍用于读取规则，有回退逻辑）
- `ExecutorPrivateKey` 升级为**启动必填**（合约不能再兜底）
- 后端钱包需持有 USDC（以前由雇主授权给合约，现在直接由后端钱包持有）

## 风险评级

| 风险项 | 等级 | 说明 |
|---|---|---|
| 后端钱包单点故障 | 🔴 High | 后端钱包私钥泄露 = 所有 USDC 资金丢失；生产环境建议迁移到 Gnosis Safe |
| 后端钱包余额耗尽 | 🟡 Medium | 需要监控 USDC 余额，余额不足时发薪失败但不影响资金安全 |
| approve 交易失败 | 🟡 Medium | 若 approve 失败，后续 LiFi 调用会直接失败并返回错误，不会丢款 |
| nonce 竞争 | 🟢 Low | 多条规则串行执行（approve 等待上链后才继续），NonceManager 串行管理 |
| LiFi API 不可用 | 🟢 Low | 返回错误，发薪记录标记 failed，不影响资金 |

## 缓解措施（已实现）

- `approveUSDC` 调用 `bind.WaitMined` 确保 approve 上链后再发 Composer 交易
- `nonceMgr.Reset()` 在所有错误路径中正确调用，防止 nonce 卡死
- 无合约配置时回退为单条 USDC 100% 规则（防止发薪静默失败）
- `ExecutorPrivateKey` 启动时校验，防止后端以空私钥启动
